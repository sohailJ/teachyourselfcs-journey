## SQL BASICS

### GROUP BY + HAVING + ORDER BY

```sql
SELECT [DISTINCT] <column expression list>
  FROM <single table>
 [WHERE <predicate>]
 [GROUP BY <column list>]
 [HAVING <predicate>]
 [ORDER BY <column list>]
 [LIMIT <integer>]
```

**Execution order:**

| Step | Clause | What it does |
| ---- | ------ | ----------- |
| 1 | `FROM` | Identify the table |
| 2 | `WHERE` | Apply selections — eliminate rows |
| 3 | `SELECT` | Project away columns — keep only those needed in SELECT, GROUP BY, HAVING |
| 4 | `GROUP BY` | Form groups & aggregate |
| 5 | `HAVING` | Eliminate groups |
| 6 | `DISTINCT` | Eliminate duplicates |
| 7 | `ORDER BY` | Sort the result |

## SET OPERATIONS: UNION, INTERSECT, EXCEPT

```text
R = {A, A, A, A, B, B, C, D}
S = {A, A, B, B, B, C, E}

UNION     → {A, B, C, D, E}
INTERSECT → {A, B, C}
EXCEPT    → {D}

```

Set operations can be used with the ALL key word (Multiset Semantics)

**UNION ALL** — sum of cardinalities:

```text
R = {A, A, A, A, B, B, C, D} = {A(4), B(2), C(1), D(1)}
S = {A, A, B, B, B, C, E}    = {A(2), B(3), C(1), E(1)}

UNION ALL → {A(4+2), B(2+3), C(1+1), D(1+0), E(0+1)}
          = {A, A, A, A, A, A, B, B, B, B, B, C, C, D, E}

```

**INTERSECT ALL** — min of cardinalities:

```text
INTERSECT ALL → {A(min(4,2)), B(min(2,3)), C(min(1,1)), D(min(1,0)), E(min(0,1))}
              = {A, A, B, B, C}
```

**EXCEPT ALL** — max(0, count_R - count_S):

```text
EXCEPT ALL → {A(max(0,4-2)), B(max(0,2-3)), C(max(0,1-1)), D(max(0,1-0)), E(max(0,0-1))}
           = {A, A, D}
```

## NESTED QUERIES

### IN and NOT IN

```sql
SELECT *
FROM table1 T1
WHERE T1.name IN (
    SELECT T2.name
    FROM table2 T2
);
```

### EXISTS and NOT EXISTS

```sql
SELECT *
FROM table1 T1
WHERE EXISTS (
    SELECT 1
    FROM table2 T2
    WHERE T1.name = T2.name
);
```

Returns all rows from table1 if and only if at least one matching row exists in table2 (matched on name).

doesn't make any sens here but just for the sake of the example and is a LEGAL query

IMPORTANT NOTE: in the example above the subquery is correlated. for each tuple of T1 the subquery is recomputed

### ANY and ALL

```sql
-- Find sailors whose rating is greater than that of *some* sailor called Popeye
SELECT *
FROM   Sailors S
WHERE  S.rating > ANY (
           SELECT  S2.rating
           FROM    Sailors S2
           WHERE   S2.sname = 'Popeye'
       );
```

**The sailor with the highest rating — Correct or Incorrect? Same or different?**

```sql
-- Version 1: using ALL
SELECT *
FROM   Sailors S
WHERE  S.rating >= ALL (
           SELECT S2.rating
           FROM   Sailors S2
       );

-- Version 2: using MAX
SELECT *
FROM   Sailors S
WHERE  S.rating = (
           SELECT MAX(S2.rating)
           FROM   Sailors S2
       );
```

Both are correct and return the same result. `>= ALL` and `= MAX(...)` are equivalent.

**But what about this version?**

```sql
-- Version 3: using ORDER BY + LIMIT
SELECT *
FROM   Sailors S
ORDER BY rating DESC
LIMIT 1;
```

**NOT equivalent** — `LIMIT 1` returns only one row even if multiple sailors share the highest rating (ties broken arbitrarily). Versions 1 and 2 return all sailors tied for the top.

credit: CS186Berkeley

### DIVISION

Relational Division: "Find sailors who've reserved all boats."

Said differently: "sailors with no counterexample missing boats"

           FROM   Boats B
           WHERE  NOT EXISTS (
                      SELECT R.bid
                      FROM   Reserves R

### ARGMAX

Find the row(s) with the maximum value of a column.

```sql
-- Version 1: using ALL
SELECT *
FROM   Sailors S
WHERE  S.rating >= ALL (
           SELECT S2.rating
           FROM   Sailors S2
       );

-- Version 2: using MAX (equivalent)
SELECT *
FROM   Sailors S
WHERE  S.rating = (
           SELECT MAX(S2.rating)
           FROM   Sailors S2
       );
```

Both return **all** sailors tied for the highest rating.

```sql
-- Version 3: using ORDER BY + LIMIT — NOT equivalent
SELECT *
FROM   Sailors S
ORDER BY rating DESC
LIMIT 1;
```

`LIMIT 1` returns only one row even when multiple sailors share the top rating (tie broken arbitrarily).

## JOINS

### Sample tables

#### R

| id | rval |
| -- | ---- |
| 1 | A |
| 2 | B |
| 3 | C |

#### S

| id | sval |
| -- | ---- |
| 2 | X |
| 3 | Y |
| 4 | Z |

R has id 1,2,3 — S has id 2,3,4. Only id=2 and id=3 match. The non-key columns are named differently (`rval`, `sval`) so NATURAL JOIN only joins on `id`.

---

### CROSS JOIN (Cartesian Product)

Returns every combination of rows — no condition. |R| × |S| = 3 × 3 = **9 rows**.

```sql
SELECT * FROM R CROSS JOIN S;
```

| R.id | R.rval | S.id | S.sval |
| ---- | ------ | ---- | ------ |
| 1 | A | 2 | X |
| 1 | A | 3 | Y |
| 1 | A | 4 | Z |
| 2 | B | 2 | X |
| 2 | B | 3 | Y |
| 2 | B | 4 | Z |
| 3 | C | 2 | X |
| 3 | C | 3 | Y |
| 3 | C | 4 | Z |

No join condition — every R row is paired with every S row regardless of any relationship.

### INNER JOIN

Returns only rows where the condition is satisfied. Unmatched rows on either side are dropped.

```sql
SELECT * FROM R INNER JOIN S ON R.id = S.id;
```

| R.id | R.rval | S.id | S.sval |
| ---- | ------ | ---- | ------ |
| 2 | B | 2 | X |
| 3 | C | 3 | Y |

R.id=1 dropped (no match in S). S.id=4 dropped (no match in R).

### LEFT OUTER JOIN

All rows from R; matched S columns filled in. Unmatched R rows get `NULL` for S columns.

```sql
SELECT * FROM R LEFT JOIN S ON R.id = S.id;
```

| R.id | R.rval | S.id | S.sval |
| ---- | ------ | ---- | ------ |
| 1 | A | NULL | NULL |
| 2 | B | 2 | X |
| 3 | C | 3 | Y |

R.id=1 is kept with NULLs. S.id=4 is still dropped.

### RIGHT OUTER JOIN

Mirror of LEFT JOIN — all rows from S, unmatched S rows get `NULL` for R columns.

```sql
SELECT * FROM R RIGHT JOIN S ON R.id = S.id;
```

| R.id | R.rval | S.id | S.sval |
| ---- | ------ | ---- | ------ |
| 2 | B | 2 | X |
| 3 | C | 3 | Y |
| NULL | NULL | 4 | Z |

S.id=4 is kept with NULLs. R.id=1 is dropped.

### FULL OUTER JOIN

All rows from both sides. Unmatched rows on either side get `NULL` for the other side's columns.

```sql
SELECT * FROM R FULL OUTER JOIN S ON R.id = S.id;
```

| R.id | R.rval | S.id | S.sval |
| ---- | ------ | ---- | ------ |
| 1 | A | NULL | NULL |
| 2 | B | 2 | X |
| 3 | C | 3 | Y |
| NULL | NULL | 4 | Z |

R.id=1 and S.id=4 are both kept, padded with NULLs.

> **FULL JOIN vs CROSS JOIN:** CROSS JOIN has no condition and pairs everything — it never produces NULLs. FULL JOIN has a condition, matches where it holds, and pads unmatched rows with NULLs. They are completely different operations.

### NATURAL JOIN

Implicit equi-join on **all columns with the same name**. Matching columns appear once in the result.

```sql
SELECT * FROM R NATURAL JOIN S;
-- equivalent to: SELECT R.id, R.rval, S.sval FROM R JOIN S ON R.id = S.id
```

| id | rval | sval |
| -- | ---- | ---- |
| 2 | B | X |
| 3 | C | Y |

Dangerous in practice — column renames silently change join behavior.

### Summary

| Join type | Rows kept |
| --------- | --------- |
| CROSS JOIN | All combinations — no condition, no NULLs |
| INNER JOIN | Matching rows only |
| LEFT JOIN | All of R + matching S (NULLs for non-matches) |
| RIGHT JOIN | All of S + matching R (NULLs for non-matches) |
| FULL JOIN | All rows from both (NULLs where no match) |
| NATURAL JOIN | INNER JOIN on shared column names |

## VIEWS

A view is a named query stored in the database. It behaves like a table but holds no data itself — it is re-executed every time it is referenced.

```sql
CREATE VIEW view_name AS
SELECT ...
FROM   ...
WHERE  ...;
```

### Why use views?

- **Simplify complex queries** — wrap a messy join or subquery behind a clean name.
- **Reuse** — reference the same logic from multiple queries without copy-pasting.
- **Access control** — expose only specific columns/rows to certain users.

### Example

```sql
CREATE VIEW ActiveSailors AS
SELECT sid, sname, rating
FROM   Sailors
WHERE  rating >= 8;

-- Use it just like a table:
SELECT * FROM ActiveSailors WHERE sname LIKE 'B%';
```

### Materialized Views

A regular view reruns its query on every access. A **materialized view** stores the result physically and must be refreshed explicitly.

```sql
CREATE MATERIALIZED VIEW ActiveSailors AS
SELECT sid, sname, rating
FROM   Sailors
WHERE  rating >= 8;

-- Refresh when underlying data changes:
REFRESH MATERIALIZED VIEW ActiveSailors;
```

| | View | Materialized View |
| - | ---- | ----------------- |
| Data stored? | No — query re-runs on access | Yes — snapshot stored |
| Always up to date? | Yes | No — must be refreshed |
| Fast reads? | Depends on query cost | Yes |

## CTEs (Common Table Expressions)

A CTE is a named subquery defined with `WITH` at the top of a query. It can be referenced like a table within that query.

```sql
WITH cte_name AS (
    SELECT ...
    FROM   ...
    WHERE  ...
)
SELECT *
FROM   cte_name;
```

### vs Subqueries / Views

| | Subquery | View | CTE |
| - | -------- | ---- | --- |
| Scope | Inline, one use | Persistent, DB-wide | Query-scoped, reusable within the query |
| Named | No | Yes | Yes |
| Reusable in same query | No | Yes | Yes |
| Stored in DB | No | Yes | No |

### Multiple CTEs

```sql
WITH
  cte1 AS (SELECT ...),
  cte2 AS (SELECT ... FROM cte1)
SELECT *
FROM   cte2;
```

Each CTE can reference the ones defined before it.

### Recursive CTEs

CTEs can be recursive — useful for tree/graph traversal.

```sql
WITH RECURSIVE counter(n) AS (
    SELECT 1                          -- base case
    UNION ALL
    SELECT n + 1 FROM counter WHERE n < 5  -- recursive case
)
SELECT * FROM counter;
-- returns: 1, 2, 3, 4, 5
```

Structure: `base case  UNION ALL  recursive case` — terminates when the recursive case returns no rows.

## NULLs

### Rules

- **(x op NULL) is NULL** — any arithmetic or comparison with NULL propagates NULL.
- **WHERE NULL** — a row is only included in output if the WHERE predicate evaluates to `TRUE`. `NULL` (unknown) is treated as not true → row is dropped.
- **Boolean connectives: 3-valued logic** — `TRUE`, `FALSE`, `NULL` (unknown).

| AND | TRUE | FALSE | NULL |
| --- | ---- | ----- | ---- |
| TRUE | TRUE | FALSE | NULL |
| FALSE | FALSE | FALSE | FALSE |
| NULL | NULL | FALSE | NULL |

| OR | TRUE | FALSE | NULL |
| -- | ---- | ----- | ---- |
| TRUE | TRUE | TRUE | TRUE |
| FALSE | TRUE | FALSE | NULL |
| NULL | TRUE | NULL | NULL |

| NOT | |
| --- | --- |
| TRUE | FALSE |
| FALSE | TRUE |
| NULL | NULL |

- **Aggregates ignore NULL-valued inputs** — `SUM`, `AVG`, `MIN`, `MAX`, `COUNT(col)` all skip NULLs. `COUNT(*)` counts all rows including those with NULLs.

### Checking for NULL

```sql
-- Correct:
WHERE col IS NULL
WHERE col IS NOT NULL

-- Wrong — always evaluates to NULL, never TRUE:
WHERE col = NULL
```

## WINDOW FUNCTIONS

> **Note:** Not covered in CS186 Berkeley. Added as a supplement.

Window functions compute a value for each row based on a "window" of related rows — without collapsing them into groups like `GROUP BY` does.

```sql
SELECT col,
       <window_fn>() OVER (
           [PARTITION BY <col>]
           [ORDER BY <col>]
           [ROWS/RANGE BETWEEN ...]
       )
FROM   table;
```

- `PARTITION BY` — splits rows into independent windows (like `GROUP BY` but rows are kept).
- `ORDER BY` — defines order within each window (required for ranking/cumulative functions).
- The `OVER (...)` clause is what makes it a window function, not an aggregate.

### Sample table

#### Sailors

| sid | sname | rating | bid |
| --- | ----- | ------ | --- |
| 1 | Alice | 9 | 101 |
| 2 | Bob | 7 | 101 |
| 3 | Carol | 9 | 102 |
| 4 | Dave | 5 | 102 |
| 5 | Eve | 7 | 102 |

### Ranking functions

```sql
SELECT sname, rating,
       RANK()        OVER (ORDER BY rating DESC) AS rank,
       DENSE_RANK()  OVER (ORDER BY rating DESC) AS dense_rank,
       ROW_NUMBER()  OVER (ORDER BY rating DESC) AS row_num
FROM   Sailors;
```

| sname | rating | rank | dense_rank | row_num |
| ----- | ------ | ---- | ---------- | ------- |
| Alice | 9 | 1 | 1 | 1 |
| Carol | 9 | 1 | 1 | 2 |
| Bob | 7 | 3 | 2 | 3 |
| Eve | 7 | 3 | 2 | 4 |
| Dave | 5 | 5 | 3 | 5 |

| Function | Ties | Gap after tie? |
| -------- | ---- | -------------- |
| `RANK()` | Same rank | Yes (1,1,**3**) |
| `DENSE_RANK()` | Same rank | No (1,1,**2**) |
| `ROW_NUMBER()` | Arbitrary tiebreak | N/A — always unique |

### Aggregate as window function

Any aggregate can be used as a window function — it computes the aggregate over the window without collapsing rows.

```sql
-- Running SUM of rating, partitioned by boat, ordered by sid
SELECT sname, bid, rating,
       SUM(rating) OVER (PARTITION BY bid ORDER BY sid) AS running_total
FROM   Sailors;
```

| sname | bid | rating | running_total |
| ----- | --- | ------ | ------------- |
| Alice | 101 | 9 | 9 |
| Bob | 101 | 7 | 16 |
| Carol | 102 | 9 | 9 |
| Dave | 102 | 5 | 14 |
| Eve | 102 | 7 | 21 |

Each partition (bid=101, bid=102) resets independently.

### NTILE

Divides rows into N roughly equal buckets.

```sql
SELECT sname, rating,
       NTILE(3) OVER (ORDER BY rating DESC) AS bucket
FROM   Sailors;
```

| sname | rating | bucket |
| ----- | ------ | ------ |
| Alice | 9 | 1 |
| Carol | 9 | 1 |
| Bob | 7 | 2 |
| Eve | 7 | 2 |
| Dave | 5 | 3 |

### LAG / LEAD

Access a value from a previous or following row.

```sql
SELECT sname, rating,
       LAG(rating,  1) OVER (ORDER BY rating) AS prev_rating,
       LEAD(rating, 1) OVER (ORDER BY rating) AS next_rating
FROM   Sailors;
```

| sname | rating | prev_rating | next_rating |
| ----- | ------ | ----------- | ----------- |
| Dave | 5 | NULL | 7 |
| Bob | 7 | 5 | 7 |
| Eve | 7 | 7 | 9 |
| Alice | 9 | 7 | 9 |
| Carol | 9 | 9 | NULL |

`NULL` at the boundaries — no previous row for the first, no next row for the last.

### Key distinction vs GROUP BY

`GROUP BY` collapses rows — you lose individual row identity. Window functions keep every row and attach the computed value alongside it.

```sql
-- GROUP BY: one row per boat
SELECT bid, AVG(rating) FROM Sailors GROUP BY bid;
```

| bid | avg |
| --- | --- |
| 101 | 8.0 |
| 102 | 7.0 |

```sql
-- Window: every sailor kept, boat avg attached
SELECT sname, bid, rating,
       AVG(rating) OVER (PARTITION BY bid) AS avg_for_boat
FROM   Sailors;
```

| sname | bid | rating | avg_for_boat |
| ----- | --- | ------ | ------------ |
| Alice | 101 | 9 | 8.0 |
| Bob | 101 | 7 | 8.0 |
| Carol | 102 | 9 | 7.0 |
| Dave | 102 | 5 | 7.0 |
| Eve | 102 | 7 | 7.0 |
