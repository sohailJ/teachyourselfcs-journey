## SQL BASICS

### GROUP BY + HAVING + ORDER BY

```sql
SELECT [DISTINCT] <column expression list>
  FROM <single table>
 [WHERE <predicate>]
 [GROUP BY <column list>]
 [HAVING predicate>]
 [ORDER BY <column list>]
 [LIMIT <integer>]
```

**Execution order:**

| Step | Clause | What it does |
| ---- | ------ | ----------- |
| 1 | `FROM` | Identify the table |
| 2 | `WHERE` | Apply selections — eliminate rows |
| 3 | `SELECT` | Project away columns — keep only those needed in SELECT, GROUP BY, HAVING |
| 4 | `GROUP BY` | Form groups & aggregate |
| 5 | `HAVING` | Eliminate groups |
| 6 | `DISTINCT` | Eliminate duplicates |
| 7 | `ORDER BY` | Sort the result |

## SET OPERATIONS: UNION, INTERSECT, EXCEPT

```text
R = {A, A, A, A, B, B, C, D}
S = {A, A, B, B, B, C, E}

UNION     → {A, B, C, D, E}
INTERSECT → {A, B, C}
EXCEPT    → {D}

```

Set operations can be used with the ALL key word (Multiset Semantics)

**UNION ALL** — sum of cardinalities:

```text
R = {A, A, A, A, B, B, C, D} = {A(4), B(2), C(1), D(1)}
S = {A, A, B, B, B, C, E}    = {A(2), B(3), C(1), E(1)}

UNION ALL → {A(4+2), B(2+3), C(1+1), D(1+0), E(0+1)}
          = {A, A, A, A, A, A, B, B, B, B, B, C, C, D, E}

```

**INTERSECT ALL** — min of cardinalities:

```text
INTERSECT ALL → {A(min(4,2)), B(min(2,3)), C(min(1,1)), D(min(1,0)), E(min(0,1))}
              = {A, A, B, B, C}
```

**EXCEPT ALL** — max(0, count_R - count_S):

```text
EXCEPT ALL → {A(max(0,4-2)), B(max(0,2-3)), C(max(0,1-1)), D(max(0,1-0)), E(max(0,0-1))}
           = {A, A, D}
```

## NESTED QUERIES

### IN and NOT IN

```sql
SELECT *
FROM table1 T1
WHERE T1.name IN (
    SELECT T2.name
    FROM table2 T2
);
```

### EXISTS and NOT EXISTS

```sql
SELECT *
FROM table1 T1
WHERE EXISTS (
    SELECT 1
    FROM table2 T2
    WHERE T1.name = T2.name
);
```

Returns all rows from table1 if and only if at least one matching row exists in table2 (matched on name).

doesn't make any sens here but just for the sake of the example and is a LEGAL query

IMPORTANT NOTE: in the example above the subquery is correlated. for each tuple of T1 the subquery is recomputed

### ANY and ALL

```sql
-- Find sailors whose rating is greater than that of *some* sailor called Popeye
SELECT *
FROM   Sailors S
WHERE  S.rating > ANY (
           SELECT  S2.rating
           FROM    Sailors S2
           WHERE   S2.sname = 'Popeye'
       );
```

**The sailor with the highest rating — Correct or Incorrect? Same or different?**

```sql
-- Version 1: using ALL
SELECT *
FROM   Sailors S
WHERE  S.rating >= ALL (
           SELECT S2.rating
           FROM   Sailors S2
       );

-- Version 2: using MAX
SELECT *
FROM   Sailors S
WHERE  S.rating = (
           SELECT MAX(S2.rating)
           FROM   Sailors S2
       );
```

Both are correct and return the same result. `>= ALL` and `= MAX(...)` are equivalent.

**But what about this version?**

```sql
-- Version 3: using ORDER BY + LIMIT
SELECT *
FROM   Sailors S
ORDER BY rating DESC
LIMIT 1;
```

**NOT equivalent** — `LIMIT 1` returns only one row even if multiple sailors share the highest rating (ties broken arbitrarily). Versions 1 and 2 return all sailors tied for the top.

### DIVISION

Relational Division: "Find sailors who've reserved all boats."

Said differently: "sailors with no counterexample missing boats"

```sql
SELECT S.sname
FROM   Sailors S
WHERE  NOT EXISTS (
           SELECT B.bid
           FROM   Boats B
           WHERE  NOT EXISTS (
                      SELECT R.bid
                      FROM   Reserves R
                      WHERE  R.bid = B.bid
                      AND    R.sid = S.sid
                  )
       );

-- CREDIT: CS186Berkeley
```
